## Introduction
The Aurora DSQL EF Core adapter integrates Entity Framework Core with Aurora DSQL, enabling .NET applications to use EF Core ORM features with Aurora DSQL’s distributed, highly available architecture.

## Prerequisites

- .NET 8.0+
- Entity Framework Core 9.0.7+
- Amazon Aurora DSQL cluster
- Amazon.AuroraDsql.Npgsql 1.1.0+

## Setup

```bash
dotnet add package Amazon.AuroraDsql.EntityFrameworkCore
```

Configure your `DbContext` with the `UseDsql()` extension method.

**With DI (ASP.NET Core):**
```csharp
using Amazon.AuroraDsql.EntityFrameworkCore.Extensions;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDsqlDataSource(
    "your-cluster.dsql.us-east-1.on.aws");
builder.Services.AddDbContext<MyDbContext>(
    (sp, options) => options.UseDsql(sp));
```

**Without DI:**
```csharp
using Amazon.AuroraDsql.Npgsql;
using Amazon.AuroraDsql.EntityFrameworkCore.Extensions;

var config = new DsqlConfig
{
    Host = "your-cluster.dsql.us-east-1.on.aws"
};
var dataSource = await DsqlDataSource.CreateAsync(config);

var options = new DbContextOptionsBuilder<MyDbContext>()
    .UseDsql(dataSource)
    .Options;

await using var context = new MyDbContext(options);
```

**DbContext:**
```csharp
public class MyDbContext : DbContext
{
    public MyDbContext(
        DbContextOptions<MyDbContext> options)
        : base(options) { }

    public DbSet<Product> Products { get; set; }
}
```

See [EF Core configuration docs](https://learn.microsoft.com/ef/core/dbcontext-configuration/)
for more options.

### Concurrency Control

DSQL uses Optimistic Concurrency Control (OCC). Conflicting transactions
are rejected with SqlState `40001`. The adapter includes
`DsqlExecutionStrategy` which provides automatic OCC retry with
exponential backoff and jitter.

When retries are exhausted, `RetryLimitExceededException` is thrown
wrapping the final `PostgresException`.

**Implicit transactions** — `SaveChangesAsync` is automatically retried:
```csharp
await context.SaveChangesAsync();
```

**Explicit transactions** — use `ExecuteInTransactionAsync` to get
OCC retry:
```csharp
var strategy = context.Database.CreateExecutionStrategy();
var orderId = Guid.NewGuid();

await strategy.ExecuteInTransactionAsync(
    context,
    async (ctx, ct) =>
    {
        // Required: clear stale tracked entities from any failed prior
        // attempt to avoid duplicate inserts or constraint violations
        // on retry.
        ctx.ChangeTracker.Clear();
        // ... your operations ...
        await ctx.SaveChangesAsync(ct);
    },
    async (ctx, ct) =>
    {
        // Verify function — confirms the transaction committed
        return await ctx.Orders.AsNoTracking()
            .AnyAsync(o => o.Id == orderId, ct);
    });
```

> **Note:** Unlike SQL Server's execution strategy, `SaveChangesAsync`
> inside an explicit transaction does not throw — it executes without
> retry. Use `ExecuteInTransactionAsync` for full OCC retry support.

### Primary Keys

UUID primary keys automatically get `gen_random_uuid()` defaults —
no configuration needed:

```csharp
public class Product
{
    public Guid Id { get; set; }  // auto-generated by DSQL
    public string Name { get; set; }
}
```

Your table DDL must include the matching default:
```sql
CREATE TABLE "Products" (
    "Id" uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    "Name" character varying(200) NOT NULL
);
```

To use BIGINT IDENTITY columns for auto-incrementing keys, enable
them in the options builder:

```csharp
builder.Services.AddDbContext<MyDbContext>(
    (sp, options) => options.UseDsql(sp, dsql =>
        dsql.EnableIdentityColumns()));
```

With a custom cache size (default is 65,536):
```csharp
options.UseDsql(sp, dsql =>
    dsql.EnableIdentityColumns(cacheSize: 1000));
```

Your entity must use `long` for the key:
```csharp
public class Order
{
    public long Id { get; set; }  // BIGINT IDENTITY
    public string Status { get; set; }
}
```

Table DDL:
```sql
CREATE TABLE "Orders" (
    "Id" bigint PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY
        (CACHE 1000),
    "Status" character varying(50) NOT NULL
);
```

## Features

- **Command Suppression** — Filters incompatible commands
  (`SET TRANSACTION ISOLATION LEVEL`, `SAVEPOINT`, `LOCK TABLE`)
  at the ADO.NET layer. Logged at Debug level.

- **Transaction Isolation** — Manages `BEGIN`/`COMMIT`/`ROLLBACK`
  directly, reports `IsolationLevel.RepeatableRead`.
  Other levels are logged and handled gracefully.

- **Connection Pooling** — Npgsql pooling via DbDataSource.

- **IAM Authentication** — Via `Amazon.AuroraDsql.Npgsql`.

- **Batch Operations** — Multi-entity `SaveChanges` works
  seamlessly; savepoints suppressed transparently.

### Design Considerations

- **OCC Conflicts** — SqlState `40001`. Handled automatically
  by `DsqlExecutionStrategy` (see [Concurrency Control](#concurrency-control)).

- **Referential Integrity** — DSQL delegates FK enforcement to
  the application layer. Navigation properties, `Include`,
  `ThenInclude`, and LINQ joins work normally — enforce
  consistency in your service layer.

- **Primary Keys** — See [Primary Keys](#primary-keys) above.

- **Migrations** — See [Migrations](#migrations) above.

Requires a DSQL cluster and AWS credentials
(see [Authentication](#authentication-issues)).


- CRUD (single and batch)
- Queries (Where, OrderBy, GroupBy, Distinct, Skip/Take)
- Transactions (Begin, Commit, Rollback)
- Execution strategy (OCC retry, max retries exceeded)

- Raw SQL
- Constraints (PK violations, NOT NULL)

## Troubleshooting

### Authentication Issues

Credentials resolve via the AWS SDK default chain:

1. Environment variables (`AWS_ACCESS_KEY_ID`,
   `AWS_SECRET_ACCESS_KEY`)
2. Named profile (`~/.aws/credentials`)
3. IAM role (EC2, ECS, Lambda)

Environment variables:
```bash
export AWS_ACCESS_KEY_ID=your-key
export AWS_SECRET_ACCESS_KEY=your-secret
```

Named profile:
```csharp
services.AddDsqlDataSource(
    "your-cluster.dsql.us-east-1.on.aws",
    config => config.Profile = "AwsProfile");
```

### Debug Logging

```csharp
builder.Services.AddLogging(logging =>
{
    logging.AddConsole();
    logging.SetMinimumLevel(LogLevel.Debug);
    logging.AddFilter(
        "Amazon.AuroraDsql.EntityFrameworkCore",
        LogLevel.Debug);
    logging.AddFilter(
        "Microsoft.EntityFrameworkCore.Database.Transaction",
        LogLevel.Debug);
    logging.AddFilter("Microsoft", LogLevel.Warning);
    logging.AddFilter("Npgsql", LogLevel.Warning);
});
```

> `SetMinimumLevel(LogLevel.Debug)` sets the global floor.
> Category filters narrow above it. The `Microsoft` and `Npgsql`
> filters suppress noise while allowing adapter logs through.

Example output:
```
dbug: Amazon.AuroraDsql.EntityFrameworkCore.Command[100001]
      Suppressing DSQL-incompatible command: SAVEPOINT
warn: Amazon.AuroraDsql.EntityFrameworkCore.Transaction[100002]
      DSQL uses fixed OCC isolation.
      Requested isolation level Serializable is ignored.
```
